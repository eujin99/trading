# 급등주 자동매매 프로젝트 요구사항분석서

## 1. 문서 목적

본 문서는 `eujin99/trading` 급등주 자동매매 프로젝트의 현재 구조를 분석하고, 실전 자동매매 시스템으로 발전시키기 위한 문제점, 개선 방향, 기능 요구사항, 비기능 요구사항, 리스크 관리 요구사항을 정의한다.

본 프로젝트의 최종 목표는 "항상 수익을 보장하는 시스템"이 아니라, 시장에서 반복적으로 발생하는 급등주 모멘텀 구간을 통계적으로 포착하면서도 손실폭, 연속손실, 과매매, 장애 리스크, API 오류, 추격매수 리스크를 최대한 낮추는 자동매매 시스템을 구축하는 것이다.

---

## 2. 현재 프로젝트 요약

### 2.1 현재 구조

현재 프로젝트는 한국투자증권 KIS API를 기반으로 국내/미국 주식의 급등주 후보를 스크리닝하고, 후보 점수화 후 매수, 보유 종목 점검, 조건 매도, 강제 매도 등을 수행하는 Python 기반 자동매매 봇이다.

현재 주요 구성은 다음과 같다.

| 구분 | 현재 역할 |
| --- | --- |
| `autotrader.py` | 메인 실행 엔진, 메뉴 처리, 스크리닝, 매수, 매도, 포트폴리오 상태 관리 |
| `kis_api.py` | KIS 인증, 시세 조회, 주문 가능 금액 조회, 매수/매도 주문 API 래퍼 |
| `config.py` | 계좌, API 키, 투자 비율, 손절/익절, 스크리닝 기준값 설정 |
| `config_kr.py` | 국내장 전용 설정 |
| `config_acc2.py` | 2번 계좌 전용 설정 |
| `secret_crypto.py` | 민감정보 암호화/복호화 |
| `encrypt_secret.py` | 민감정보 암호화 유틸리티 |
| `sim/` | 시뮬레이션 관련 디렉터리 |

### 2.2 현재 매매 흐름

현재 시스템의 기본 흐름은 다음과 같다.

1. 사용자가 메뉴에서 스크리닝 실행
2. 거래량 상위/등락률 상위 종목 수집
3. 후보 종목을 점수화
4. 점수 상위 종목 매수
5. 보유 종목 상태를 메모리에 저장
6. 사용자가 메뉴에서 보유 종목 점검 실행
7. 목표가/손절가 조건에 따라 일부 또는 전량 매도
8. 필요 시 텔레그램으로 보유 상태와 매도 명령 처리

## 3. 핵심 문제점 진단

### 3.1 자동매매가 아니라 수동 트리거형 구조

현재 가장 큰 문제는 손절/익절/포트폴리오 점검이 완전 자동 주기 실행이 아니라 사용자가 메뉴를 직접 실행해야 한다는 점이다.

급등주는 수 분 안에 수익과 손실이 크게 바뀐다. 따라서 수동 점검 구조에서는 다음 문제가 발생한다.

| 문제 | 위험 |
| --- | --- |
| 사용자가 메뉴 4를 늦게 실행 | 손절 지연 |
| API 오류 발생 후 재시도 부재 | 포지션 방치 |
| 장중 급락 발생 | 손실 확대 |
| 목표가 도달 후 되밀림 | 수익 반납 |
| 강제 청산 시간 미작동 | 익일 갭하락 노출 |

#### 개선 방향

반드시 이벤트 루프 기반 자동 실행 구조로 전환해야 한다.

필수 자동 루프는 다음과 같다.

1. 장전 관찰 루프
2. 장중 스크리닝 루프
3. 포지션 감시 루프
4. 주문 체결 확인 루프
5. 리스크 컷 루프
6. 장마감 강제 정리 루프
7. 장애 감시 및 텔레그램 긴급 알림 루프

### 3.2 급등주 스코어링이 단순함

현재 스코어링은 주로 거래량, 등락률, 52주 신고가 근접, 52주 저점 대비 반등, PER 정도를 사용한다.

문제는 급등주 매매에서 중요한 다음 요소들이 부족하다는 점이다.

| 부족 요소 | 필요한 이유 |
| --- | --- |
| 분봉 추세 | 급등이 유지 중인지, 이미 꺾였는지 판단 |
| 체결강도 | 실제 매수세 유입 여부 판단 |
| 호가 잔량 | 매도벽/매수벽 판단 |
| 거래대금 증가율 | 소형주 허수 거래량 필터링 |
| 시가 대비 위치 | 장초반 고점 추격 방지 |
| VWAP 이탈 여부 | 기관/단타 기준선 이탈 감지 |
| VI/투자경고/관리종목 필터 | 급락/거래정지 리스크 회피 |
| 뉴스/공시 이벤트 | 테마성 급등과 허위 급등 구분 |
| 갭상승률 | 과도한 갭상승 후 되밀림 방지 |
| 변동성 기반 손절폭 | 종목별 변동성 차이 반영 |

#### 개선 방향

기존 점수제는 유지하되, 아래 구조로 바꿔야 한다.

1. 1차 필터: 거래 가능 종목 필터
2. 2차 필터: 유동성/거래대금 필터
3. 3차 필터: 과열/위험 종목 제거
4. 4차 필터: 모멘텀 지속성 점수
5. 5차 필터: 진입 타이밍 점수
6. 6차 필터: 리스크 대비 기대수익 점수
7. 최종 진입 여부: 점수 + 시장상태 + 포지션 리스크 동시 확인

### 3.3 물타기 로직은 급등주 전략과 충돌 가능성이 높음

현재 시스템에는 손절 구간 진입 시 1회 물타기 로직이 존재한다. 급등주 매매에서 물타기는 매우 위험하다.

급등주는 추세가 무너지면 하락 속도가 빠르고, 수급이 빠져나가면 반등 없이 급락할 수 있다. 이때 손실 포지션에 추가매수하면 평균단가는 낮아지지만 총손실금액과 계좌 리스크는 커진다.

#### 개선 방향

물타기는 기본 비활성화해야 한다.

대신 아래 구조를 사용한다.

| 기존 방식 | 개선 방식 |
| --- | --- |
| 손실 구간 물타기 | 손실 구간 즉시 축소 또는 청산 |
| 평단 낮추기 | 리스크 낮추기 |
| 추가매수 | 수익 구간 피라미딩 |
| 단순 손절률 | ATR/VWAP/분봉 저점 기반 트레일링 스탑 |

예외적으로 물타기를 허용하려면 아래 조건을 모두 만족해야 한다.

1. 시장지수 상승
2. 해당 종목 VWAP 위 회복
3. 체결강도 회복
4. 1분봉 또는 3분봉 저점 재돌파
5. 추가매수 후 총 리스크가 계좌의 0.3~0.5% 이내
6. 당일 동일 종목 재진입 횟수 제한
7. 자동 청산 조건 강화

### 3.4 실시간 리스크 관리가 부족함

현재 시스템은 종목별 손절가와 목표가가 있으나, 계좌 전체 리스크 관리가 부족하다.

필수로 필요한 계좌 단위 리스크 제한은 다음과 같다.

| 항목 | 권장 기준 |
| --- | --- |
| 1회 거래 최대 손실 | 총자산의 0.3~0.7% |
| 1일 최대 손실 | 총자산의 1.0~2.0% |
| 1일 최대 거래 횟수 | 3~8회 |
| 동일 종목 재진입 | 1일 1회 또는 최대 2회 |
| 연속 손실 제한 | 2~3회 손실 시 당일 중단 |
| 최대 동시 보유 종목 | 2~3개 |
| 종목당 최대 비중 | 총자산의 10~20% |
| 현금 최소 비중 | 30~50% |
| 장마감 보유 여부 | 전략 검증 전 원칙적으로 당일 청산 |

#### 개선 방향

다음 리스크 모듈을 별도 파일로 분리한다.

`risk_manager.py`

역할:

1. 주문 전 리스크 승인
2. 주문 후 포지션 리스크 계산
3. 계좌 전체 손익 감시
4. 일일 손실 제한
5. 연속 손실 제한
6. API 오류 시 신규매수 차단
7. 장마감 강제 청산
8. 과매매 방지
9. 종목별 최대 손실금액 계산
10. 텔레그램 긴급 중단 명령 처리

### 3.5 백테스트/시뮬레이션 검증 체계가 부족함

현재 프로젝트는 실시간 API 기반 실행 코드가 중심이다. 하지만 자동매매 프로젝트에서 실전 투입 전 가장 중요한 것은 과거 데이터 기반 검증이다.

현재 필요한 검증은 다음과 같다.

| 검증 항목 | 설명 |
| --- | --- |
| 과거 급등주 재현성 | 과거 급등 종목에서 신호가 실제로 발생했는지 |
| 진입 후 최대 상승률 | 진입 후 목표가 도달 가능성 |
| 진입 후 최대 하락률 | 손절폭 적정성 |
| MFE/MAE 분석 | 최대 유리/불리 움직임 분석 |
| 체결 가능성 | 호가/거래대금 기준 실제 체결 가능 여부 |
| 슬리피지 반영 | 시장가 체결 시 불리한 가격 반영 |
| 수수료/세금 반영 | 실제 순손익 계산 |
| 장세별 성능 | 상승장/하락장/횡보장 분리 |
| 시간대별 성능 | 09:00~09:30, 09:30~10:30, 오후장 구분 |
| 종목군별 성능 | 대형주/중형주/소형주/테마주 구분 |

#### 개선 방향

`backtest/` 디렉터리를 신설한다.

필수 파일:

| 파일 | 역할 |
| --- | --- |
| `data_loader.py` | 과거 OHLCV/분봉 데이터 로드 |
| `strategy.py` | 실전과 동일한 신호 로직 |
| `simulator.py` | 주문/체결/수수료/슬리피지 시뮬레이션 |
| `metrics.py` | 승률, 기대값, MDD, 샤프, 손익비 계산 |
| `walk_forward.py` | 구간별 검증 |
| `report.py` | 결과 리포트 생성 |

### 3.6 포트폴리오 상태 저장이 메모리 중심

현재 보유 종목 상태는 `portfolio` 딕셔너리 중심으로 관리된다. 일부 일일 손익/거래 가드는 JSON 파일로 저장되지만, 포지션 전체 상태와 주문 이력, 신호 이력, 체결 이력의 영속 저장 구조가 부족하다.

문제는 다음과 같다.

| 문제 | 위험 |
| --- | --- |
| 프로그램 종료 | 내부 포지션 상태 손실 |
| 수동매매와 봇 상태 불일치 | 잘못된 수량 매도 가능 |
| 부분체결 | 평균단가/잔량 계산 오류 |
| API 응답 실패 | 실제 계좌와 내부 상태 불일치 |
| 재시작 | 손절/익절 조건 초기화 |

#### 개선 방향

SQLite 기반 로컬 DB를 도입한다.

필수 테이블:

1. `trades`
2. `orders`
3. `positions`
4. `signals`
5. `risk_events`
6. `daily_pnl`
7. `system_logs`
8. `strategy_params`

### 3.7 주문 체결 관리가 부족함

현재 주문은 시장가 중심이다. 급등주에서는 시장가 주문이 빠르다는 장점이 있지만, 호가가 얇은 종목에서는 슬리피지가 커진다.

필수 개선 사항은 다음과 같다.

1. 주문 요청
2. 주문번호 저장
3. 체결 여부 확인
4. 부분체결 처리
5. 미체결 취소
6. 체결가 기준 포지션 업데이트
7. 슬리피지 계산
8. 주문 실패 재시도
9. API 오류 시 신규 주문 차단
10. 텔레그램 긴급 알림

## 5. 기능 요구사항

### 5.1 스크리닝 기능

#### FR-001. 장중 급등주 후보 수집

시스템은 장중 일정 주기마다 급등주 후보를 자동 수집해야 한다.

필수 조건:

1. 거래량 상위
2. 등락률 상위
3. 거래대금 상위
4. 체결강도 상위
5. 시가 대비 상승률
6. 전일 대비 상승률
7. 분봉 거래량 증가율

#### FR-002. 위험 종목 필터링

시스템은 다음 종목을 자동 제외해야 한다.

1. ETF
2. ETN
3. 스팩
4. 우선주
5. 관리종목
6. 투자경고/투자위험 종목
7. 거래정지 위험 종목
8. 거래대금 부족 종목
9. 호가 스프레드 과도 종목
10. 당일 이미 손절한 종목
11. 과도한 갭상승 종목
12. 급등 후 VWAP 이탈 종목

#### FR-003. 다중 점수화 모델

시스템은 후보 종목에 대해 다음 점수를 계산해야 한다.

| 점수 | 설명 |
| --- | --- |
| Volume Score | 거래량/거래대금 증가 |
| Momentum Score | 상승률, 분봉 추세 |
| Liquidity Score | 체결 가능성 |
| Stability Score | 변동성 안정성 |
| Timing Score | 진입 타이밍 |
| Risk Score | 손절폭 대비 기대수익 |
| Market Score | 시장지수 방향성 |
| News/Event Score | 공시/뉴스 모멘텀 |

최종 점수는 단순 합산이 아니라 리스크 가중치를 반영해야 한다.

### 5.2 진입 기능

#### FR-004. 주문 전 리스크 승인

모든 매수 주문은 `risk_manager`의 승인을 받아야 한다.

승인 조건:

1. 당일 손실 한도 미초과
2. 연속 손실 제한 미초과
3. 종목당 최대 비중 미초과
4. 총 투자 비중 미초과
5. 현금 비중 유지
6. 동일 종목 재진입 제한
7. 장마감 임박 시간 신규매수 금지
8. API 상태 정상
9. 스프레드 허용 범위 이내
10. 예상 손실금액이 계좌 허용 리스크 이내

#### FR-005. 포지션 사이징

시스템은 고정 금액이 아니라 손절폭 기반 수량 계산을 해야 한다.

기본 공식:

```text
허용손실금액 = 총자산 x 거래당 리스크 비율
1주당 위험금액 = 진입가 - 손절가
매수수량 = 허용손실금액 / 1주당 위험금액
```

예시:

```text
총자산: 10,000,000원
거래당 리스크: 0.5%
허용손실금액: 50,000원
진입가: 10,000원
손절가: 9,700원
1주당 위험금액: 300원
매수수량: 166주
```

## 11. 최종 결론

현재 프로젝트는 급등주 자동매매의 기본 골격은 갖추고 있으나, 실전 자동매매 시스템으로 보기에는 아직 위험하다.

가장 치명적인 문제는 다음과 같다.

1. 손절/익절이 완전 자동이 아님
2. 포지션 상태 저장이 불완전함
3. 물타기 로직이 급등주 전략과 충돌함
4. 주문 체결/부분체결 관리가 부족함
5. 백테스트 검증 체계가 부족함
6. 계좌 단위 리스크 제한이 부족함
7. 스코어링이 급등주 특유의 위험을 충분히 반영하지 못함

따라서 실전 투입 전 최우선 목표는 수익률 극대화가 아니라 다음 세 가지다.

1. 손실 제한 자동화
2. 주문/포지션 상태 복구 가능성 확보
3. 검증 가능한 전략 구조 구축

이 세 가지가 완성된 후에야 급등주 매매 전략의 수익률 개선을 논할 수 있다.
